Log Normal

This will be all the math involved with the Log Normal (LN?) trading function.

Conceptual Overview

The normal strategy provides the LP with a a log-normal shaped liquidity distribution centered around a price K with a width given by σ. This strategy can be made time-dependent by an additional τ parameter that is the time til the pool will "expire". In this case, the LN trading function provides the LP with a payoff that is equivalent to a Black-Scholes covered call option with strike K, implied volatility σ, and time to expiration τ. The parameters K and σ can also be made time dependent.

Core

LN has three variable parameters:

  • K≡strike
  • σ≡volatility
  • τ≡time_to_expiration
  • These parameters must satisfy: K>0σ>0τ>0 Let's separate variables from parameters in function inputs with ; just to stay clear (though it may be verbose). The trading function for this strategy is given by φ(x,y,L;K,σ,τ)=Φ−1(Lx​)+Φ−1(KLy​)+στ​.​​ In the equation above, x and y are reserves, and L is the liquidity. We can put: L≡liquidity Note that L has units of Token (this is what we want). Given the domain of Φ−1 we can see that x∈[0,L] and y∈[0,KL]. As the pool's liquidity increases, the maximal amount of each reserve increases and both are scaled by the same factor (this is what we want).

Useful Notation

We will use the following notation: d1​(S;K,σ,τ)=στ​lnKS​+21​σ2τ​​​ d2​(S;K,σ,τ)=στ​lnKS​−21​σ2τ​​​

Price

If we compute the derivatives and simplify the expression, we get that the pool price is given by either: PX​(x,L;K,σ,τ)=Kexp(Φ−1(1−Lx​)στ​−21​σ2τ)​​​ PY​(y,L;K,σ,τ)=Kexp(Φ−1(KLy​)στ​+21​σ2τ)​​​ Do not that other DFMMs such as the geometric mean market maker have a price that can be determined from both reserves at once so we typically do not write P_X and P_Y.

Determining L from Price and Reserves

There are a few distinct times where we need to determine the value of L, but they all come down to liquidity being deposited into the pool and not from swaps. We want to disentangle swaps and liquidity provision/donation. Note that for G3M, we don't have this same need as the L is determined by the trading function explicitly.

L from x

Without showing all the work, we can recall that Lx​ is one of the option binaries: Lx​=1−Φ((d1​(S;K,σ,τ))​​ Since we know x and we know S, we can solve for L to find: LX​(x,S;K,σ,τ)=1−Φ(d1​(S;K,σ,τ))x​​​​

L from y

The work here is basically a mirrored image of the above. KLy​=Φ(d2​(S;K,σ,τ))​​ From here we get L: LY​(y,S;K,σ,τ)=KΦ(d2​(S;K,σ,τ))y​​​​

Pool Initialization

When the pool is initialized, we need to determine the value of L. The user will provide a price S and an amount of x or an amount of y that they wish to tender. From there, we should be able to determine how much of both tokens must be allocated as well as the value of L.

Specifying x

Suppose that the user specifies the amount x0​ they wish to allocate and they also choose a price S0​. Further, we need to know how much y0​ to allocate, which we can also use the other binary in Equation (8). At this point, we know S and L and so we can get: y0​=y(x0​,S0​;K,σ,τ)=KLX​(x0​,S0​;K,σ,τ)Φ(d2​(S;K,σ,τ))​ Note that the above is not simplified and likely could be drastically simplified.

Specifying y

Suppose that the user specifies the amount y they wish to allocate and they also choose a price S. Now we need to get x: x0​=x(y0​,S0​)=LY​(y0​,S0​;K,σ,τ)(1−Φ(d1​(S;K,σ,τ)))​

Adding/Removing Liquidity

When a user adds liquidity, they will specify an amount of x or an amount of y, and the pool's price S and liquidity L will already be known. When adding liquidity, we assume that price will not change whatsoever and only the value of L will change.

Specifying x

Given some amount of δx​ the user wants to add, we can just use the equation for L(x,S) above to get: LX​(x+δx​,S;K,σ,τ)=1−Φ(d1​(S;K,σ,τ))x+δx​​ In fact, L is linear in the first variable, so: LX​(x+δx​,S;K,σ,τ)=LX​(x,S;K,σ,τ)+δL​LX​(δx​,S;K,σ,τ)​​ can be used to make the calculation easier. In fact, if we just want to determine δL​, we can just use the above equation and subtract LX​(x,S) from both sides and use our S=PX​(x) equation to get: δL,X​=δX​xL​​ where x is the amount of X reserves in the pool before the liquidity is added.

Specifying y

Given some amount of δy​ the user wants to add, we can just use the equation for L(y,S) above to get: LY​(y+δy​,S;K,σ,τ)=KΦ(d1​(S;K,σ,τ))y+δy​​ Again, L is linear in the first variable. We can use the same technique above to show that: δL,Y​=δY​yL​​​​

Getting the Inputs

Given that we have either δL,X​ or δL,Y​, we can use the above equations to get the amount of X or Y that must be added to the pool. Specifically, if the user wants to add δX​, we find δL,X​ then use Equation (10) like so: δL,X​=δY​yL​. The other way follows mutatis mutandis.

Removing Liquidity

When a user removes liquidity, they will specify an amount of x or an amount of y, and the pool's price S and liquidity L will already be known. When removing liquidity, we assume that price will not change whatsoever and only the value of L will change. We can just use the same formulation as above and note that δx​ and δy​ may be positive or negative.

Swaps

When a user swaps, it must be that the trading function φ remains invariant. Specifically, φ(x+ΔX​,y+ΔY​,L+δL​)=0. Note again I'm allowing for ΔX​ and ΔY​ to be positive or negative. In absence of fees, the liquidity L is invariant, that is δL​=0. We will continue to use the lowercase δ for fee-based changes and Δ for swap inputs and outputs to distinguish these components. To compute a swap, it is a matter of finding the ΔX​(ΔY​) or ΔY​(ΔX​) that satisfies the above equation.

With Fees

Assume now that there is a fee parameter γ such that the fee invested into the pool is 1−γ. Assume further that the fee is always taken out of the input token for the swap. Think of the swap as a two step process:

  1. Adding liquidity. E.g., δY​:=(1−γ)ΔY​. This is the amount of the input token that is added to the pool and it is what is used to calculate the change in liquidity δL​. From here, we can imagine that the swapper then takes temporary debt in adding δY​ to the pool where the δY​ can by found by using Equation (10) and we specifically get: δY​=yLδL​​
  2. Computing a no-fee swap with the remaining amount of the input token. E.g., Δx​​:=γΔx​. Note at this point, the reserves are then x+δx​ and y+δy​ and the liquidity L+δL​. So we must use these in the swap calculation. Then we can use all of the rules we defined here.

Δy​ given Δx​

Suppose that the user wants to swap x for y and the price is S. They specifically tender Δx​ and the fee parameter is γ. Now δx​=(1−γ)Δx​ and Δx​​=γΔx​. From this we get δL​=LX​(δx​,S)=1−Φ(σlnKS​+21​σ2​)δx​​ Using the trading function, we solve for Δy​: Φ−1(L+δL​x+Δx​​)+Φ−1(K(L+δL​)y+Δy​​)=−σΔy​(Δx​)=K(L+δL​)⋅Φ(−σ−Φ−1(L+δL​x+Δx​​))−y​

Δx​ given Δy​

Suppose that the user wants to swap y for x and the price is S. They specifically tender Δy​ and the fee parameter is γ. Now δy​=(1−γ)Δy​ and Δy​​=γΔy​. From this we get δL​=LY​(δy​,S)=K⋅Φ(σlnKS​−21​σ2​)δy​​ Using the trading function, we solve for Δx​: Φ−1(L+δL​x+Δx​​)+Φ−1(K(L+δL​)y+Δy​​)=−σΔx​(Δy​)=(L+δL​)⋅Φ(−σ−Φ−1(K(L+δL​)y+Δy​​))−x​

Arbitrage Math

Raising the price

When we need to raise the price, we need to tender in Y. If the current price is S and we want to raise it to S′, then we need to tender in Y such that we go from y to y′ and: y′=K⋅(L+δL​)⋅Φ(σlnKS′​−21​σ2​) and we know δL​ in terms of Δy​: K⋅Φ(σlnKS​−21​σ2​)(1−γ)Δy​​ therefore the amount of Y to tender is: Δy​=y′−y=K⋅(L+δL​)⋅Φ(σlnKS′​−21​σ2​)−y=K⋅L⋅Φ(σlnKS′​−21​σ2​)+(1−γ)Δy​⋅Φ(σlnKS​−21​σ2​)Φ(σlnKS′​−21​σ2​)​−y⟹Δy​=1+(γ−1)Φ(σlnKS​−21​σ2​)Φ(σlnKS′​−21​σ2​)​KLΦ(σlnKS′​−21​σ2​)−y​​

Lowering the price

When we need to lower the price, we need to tender in X. If the current price is S and we want to lower it to S′, then we need to tender in X such that we go from x to x′ and: Δx=(L+δL​)⋅(1−Φ(σlnKS′​+21​σ2​))−x⟹Δx​=1+(γ−1)1−Φ(σlnKS​+21​σ2​)1−Φ(σlnKS′​+21​σ2​)​L(1−Φ(σlnKS′​+21​σ2​))−x​​

Value Function on L(S)

Relate to value on V(L,S) and V(x,y). Then we can use this to tokenize. We have LX​(x,S) and LY​(y,S). We know that: V(x(S),y(S))=xS+y Now we also have the following x=LS⋅(1−Φ(σlnKS​+21​σ2​))y=K⋅L⋅Φ(σlnKS​−21​σ2​) Therefore: V(L,S)=L(S⋅(1−Φ(σlnKS​+21​σ2​))+K⋅Φ(σlnKS​−21​σ2​))​ Note that V is linear in L and so we can use this to tokenize.

Time Dependence

Note that L effectively changes as parameters of the trading function change. To see this, note that the trading function must always satisfy: Φ−1(Lx​)+Φ−1(KLy​)+στ​=0. For new parameters, K′, σ′ and τ′, we must find an L′ so that the trading function is satisfied: Φ−1(L′x​)+Φ−1(K′L′y​)+σ′τ′​=0. We can find this new L′ using a root finding algorithm.

Root Finding

We will use a bisection algorithm to determine the new L′. Suppose that (K,σ,τ,L) are the current parameters and we have (K′,σ′,τ′) as the new parameters. Then we can compute: f(L)=Φ−1(Lx​)+Φ−1(K′Ly​)+σ′τ′​.

  • If f(L)<0, then:
    • Upper bound: L
    • Lower bound: max{x,y/K}.
  • If f(L)>0, then:
    • Upper bound: L⋅1.5.
    • Lower bound: L
